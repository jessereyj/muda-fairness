CHAPTER 4

FORMAL LOGIC AND VERIFICATION

This chapter presents the formal language and semantics used to specify and verify properties of MUDA executions.

First, it defines Linear Temporal Logic (LTL) using trace semantics.

Second, it explains how the deterministic MUDA transition system from Chapter 3 induces an infinite execution trace (via terminal stuttering).

Third, it defines MUDA-specific atomic propositions as state-level predicates and explains how they are interpreted at each position of the MUDA execution trace via a fixed interpretation function.

The Rocq development mechanizes the LTL syntax and trace semantics, the MUDA-to-trace interpretation, and the mechanically checked proofs of the three fairness properties.

The Rocq development defines LTL with the temporal operators X (next), F (eventually), and G (always). The operators X and F are included in the formal syntax for completeness, but they are not used in the fairness verification results of this thesis. The fairness properties verified in this thesis are invariance properties and are stated using G only.

All fairness properties in this chapter are invariants. Each property is expressed in the form G phi, which states that phi holds at every position of the execution trace. This matches the nature of MUDA correctness, where violations must not occur at any step of the execution.

The framework is organized into three layers:

Foundation layer: LTL syntax and trace semantics (see Syntax.v and Semantics.v).
MUDA protocol layer: deterministic transition system and infinite executions via terminal stuttering (see Transitions.v and Interpretation.v).
Fairness verification layer: LTL fairness formulas and mechanically checked proofs on MUDA traces (see PriorityFairness.v, QuantityFairness.v, PriceFairness.v).

4.1 Foundation Layer

This thesis uses semantic satisfaction over a fixed, MUDA-induced trace and does not require a proof system; completeness is not required for this verification approach since satisfaction is shown directly over induced traces, rather than deriving formulas.

The fairness properties considered in this thesis are invariance properties. Therefore, they can be expressed using the temporal operator G, and no additional temporal operators are required.

4.1.1 Syntax

Let PROP be the set of atomic propositions. In this development, PROP is implemented as nat (natural-number indices). Atomic propositions are written as Atom p where p : nat (see Syntax.v).

Formulas are built from atomic propositions using propositional connectives and temporal operators. The mechanization provides X, F, and G; the fairness specifications in this thesis use G only.

Propositional: ~, /\, \/, ->
Temporal: X (next), F (eventually), G (always)

Grammar (informal):
phi ::= Atom(p) | ~phi | (phi /\ psi) | (phi \/ psi) | (phi -> psi) | X phi | F phi | G phi

These formulas are interpreted over MUDA execution traces defined below.

4.1.2 Semantics

A valuation is a function v : PROP -> Prop. An infinite trace is a sequence of valuations:
sigma = (v0, v1, v2, ...)

Satisfaction sigma, i |= phi is defined as follows:

Propositional rules:
sigma, i |= Atom(p) iff vi(p) holds
sigma, i |= ~phi iff not (sigma, i |= phi)
sigma, i |= (phi /\ psi) iff (sigma, i |= phi) and (sigma, i |= psi)
sigma, i |= (phi \/ psi) iff (sigma, i |= phi) or (sigma, i |= psi)
sigma, i |= (phi -> psi) iff not (sigma, i |= phi) or (sigma, i |= psi)

Temporal rules:
sigma, i |= X phi iff sigma, i+1 |= phi
sigma, i |= G phi iff for all j >= i, sigma, j |= phi
sigma, i |= F phi iff there exists j >= i such that sigma, j |= phi

Write sigma |= phi to mean sigma, 0 |= phi.

4.2 MUDA Protocol Layer (Executions as Infinite Traces)

This section connects the deterministic MUDA execution from Chapter 3 to the trace semantics used for LTL satisfaction.

A MUDA execution is generated by the deterministic transition function delta from Chapter 3. The deterministic transition function is written mathematically as delta : State -> State and corresponds in the mechanization to step (see Transitions.v), with delta provided as a thesis-level alias.

Given an initial state x0, the finite-step execution is defined as:
xi = execute i x0

where execute iterates delta i times.

Although matching terminates, LTL requires an infinite trace. When a terminal state is reached, the execution is extended by stuttering. If phase(x) = P7, then delta(x) = x. The execution therefore becomes an infinite sequence by repeating the terminal state.

Terminal stuttering ensures that every finite execution induces an infinite trace required for LTL semantics.

Since the transition system is deterministic, each initial state induces a unique execution trace. Therefore, verifying sigma |= phi establishes that the MUDA execution from that initial state satisfies phi globally.

4.3 Atomic Propositions and Interpretation

Atomic propositions are state-level predicates defined in Atoms.v. The mapping from PROP = nat to these predicates is defined in Interpretation.v using the function:

interp_atom : State -> nat -> Prop

The interpretation function lifts state-level predicates to trace-level formulas by evaluating each atomic predicate at every state of the execution trace.

The infinite trace is defined as:
mu(x0), where the i-th valuation is given by interp_atom (execute i x0).

The mapping includes:

p_allocOK, p_has_cprice, p_bounds_pstar, p_price_rule, p_prioB_step, p_prioS_step, and phase predicates.

Each predicate corresponds directly to definitions from Chapter 3, including allocation consistency, priority selection, and clearing price correctness.

The interpretation function correctly reflects the state-level predicates defined in Atoms.v. Therefore, satisfaction of atomic propositions corresponds directly to evaluation of these predicates on execution states.

This correspondence is formalized by the lemma mu_trace_atom_at_execute, which states that atomic propositions evaluated on the trace coincide with their evaluation on execution states.

In the remainder of this chapter, we refer to the repeated use of this correspondence together with the standard unfolding of G (formalized in the mechanization as the lemma satisfies_always_unfold) to move from state-level predicates about execute i x0 to LTL satisfaction statements about mu(x0). This step is referred to as the LTL lift step.

This lift step is used repeatedly when stating and proving the fairness invariants as LTL properties in Sections 4.4 and 4.5.

Stuttering and the lifting step play distinct roles in connecting the MUDA operational model to LTL semantics.

Stuttering ensures that every MUDA execution induces an infinite trace, as required by LTL. Formally, once phase(x) = P7 holds, delta(x) = x and the execution trace becomes a constant sequence. This avoids special termination cases when evaluating LTL formulae, and ensures that once the protocol reaches its final state, no new violations of fairness can occur.

The lifting step connects state-level predicates to trace-level satisfaction. The fairness properties are defined as predicates over execution states, such as allocOK(x) or bounds_pstar(x). However, LTL formulae are interpreted over positions of an execution trace. The lifting step uses the correspondence between execute i x0 and the i-th position of the trace mu(x0), together with the interpretation function for atomic propositions, to translate state-based statements into temporal statements. This allows properties proven over all reachable states to be expressed as LTL invariants of the form sigma |= G phi.

4.3.1 Formal Priority Correctness

Priority fairness concerns the correctness of the deterministic greedy selection step in Phase P3. This step-based notion is expressed as a predicate of the current state x by inspecting the deterministic selector computed from x.

These predicates formalize that no higher-priority feasible alternative is skipped by the deterministic selector.

Let x be a state. The two priority step predicates are defined in Atoms.v as follows.

Buyer-step priority correctness, priorityB_step_ok_prop(x), states that if phase(x) = P3 and the selector finds a feasible pair with chosen buyer b, then no feasible buyer with higher priority than b exists in x. Formally, for all buyers b' in bids(x), if prioB(b', b) holds, then there does not exist any seller a' in asks(x) such that feasibility holds for the pair (b', a') with respect to matches(x).

Seller-step priority correctness, priorityS_step_ok_prop(x), states that if phase(x) = P3 and the selector chooses the pair (b, a), then no feasible seller with higher priority than a exists for the chosen buyer b in x. Formally, for all sellers a' in asks(x), if prioS(a', a) holds and feasibility holds for (b, a'), then a contradiction follows.

4.4 Fairness Specifications (LTL)

All fairness properties are expressed as LTL formulae over the MUDA trace sigma = mu(initial_state(bs, as_list)).

4.4.1 Priority Fairness

Define:
priorityOK := (G (Atom p_prioB_step)) /\ (G (Atom p_prioS_step))

Priority fairness states that at every step where a matching decision is made, the mechanism respects the priority ordering of agents. The predicates p_prioB_step and p_prioS_step encode the correctness of the deterministic selection for buyers and sellers, respectively. The use of G ensures that this condition holds at every position along the execution trace, not only at specific matching steps. Outside Phase P3, these predicates hold trivially since no matching decision is performed.

4.4.2 Quantity Fairness

Define:
quantityOK := G (Atom p_allocOK)

Quantity fairness expresses conservation of units across the execution. The predicate p_allocOK states that, for each agent, the initial quantity is exactly decomposed into allocated and residual quantities. The temporal operator G enforces that this accounting identity holds at every state of the execution trace, including after termination where the system stutters.

Implementation note (decidable equality).
In the mechanization, the allocated quantity functions scan the match record and must test whether a match refers to a given bid or ask. This is implemented using decidable equality functions for the relevant record types (e.g., bid_eq_dec and ask_eq_dec), generated using the standard decide equality mechanism. This supports executable definitions and does not change the mathematical statement of the fairness properties.

4.4.3 Price Fairness

Define:
priceOK := G (Atom p_has_cprice -> (Atom p_bounds_pstar /\ Atom p_price_rule))

Price fairness captures the uniform pricing rule of MUDA. The predicate p_has_cprice acts as a guard that indicates when a clearing price pstar is well-defined from the current match record. When this condition holds, the clearing price must lie within the marginal bid and ask bounds, and must follow the deterministic pricing rule. The implication ensures that the pricing constraint is enforced only when a clearing price exists. The use of G guarantees that this condition holds at every step of the execution trace.

Outside the pricing-relevant conditions (e.g., when no marginal pair exists, or in phases before pricing), the corresponding atoms are defined to hold trivially; the non-trivial constraints apply exactly when a clearing price is defined.

4.5 Fairness Verification (Theorems and Proof Sketches)

Theorems are proven for all traces generated from all initial states.

Theorem 1 (Priority Fairness).
For all inputs bs and as_list, let sigma = mu(initial_state(bs, as_list)). Then:
sigma |= (G (Atom p_prioB_step)) /\ (G (Atom p_prioS_step)).

This result states that the fairness property holds at every state along the execution trace induced by any input lists bs and as_list.

Mechanized theorem: priority_fairness_LTL_initial.

Proof (sketch).
Fix an arbitrary index j >= 0 and let x = execute j (initial_state(bs, as_list)). By the interpretation lemma, satisfaction at position j reduces to evaluation of the corresponding state predicates on x. Each predicate is guarded by phase(x) = P3; outside Phase P3 it holds trivially because no matching decision is made in those states. When phase(x) = P3, the deterministic selector ensures that no feasible order with higher priority exists. Therefore, the predicates hold for every j, and the G formulas follow.

Theorem 2 (Quantity Fairness).
For all inputs bs and as_list, let sigma = mu(initial_state(bs, as_list)). Then:
sigma |= G (Atom p_allocOK).

This result states that the fairness property holds at every state along the execution trace induced by any input lists bs and as_list.

Mechanized theorem: quantity_fairness_LTL_initial.

Proof (sketch).
The predicate allocOK holds in the initial state and is preserved by each transition. Fix an arbitrary index j and consider x = execute j (initial_state(bs, as_list)). Induction over execute establishes allocOK(x) for all indices. A lemma connects allocOK to allocOK_prop, which is the interpretation of Atom p_allocOK. The lifting step yields the LTL result.

Theorem 3 (Price Fairness).
For all inputs bs and as_list, let sigma = mu(initial_state(bs, as_list)). Then:
sigma |= G (Atom p_has_cprice -> (Atom p_bounds_pstar /\ Atom p_price_rule)).

This result states that the fairness property holds at every state along the execution trace induced by any input lists bs and as_list.

Mechanized theorem: uniform_price_fairness_LTL_initial.

Proof (sketch).
Fix an arbitrary index j and let x = execute j (initial_state(bs, as_list)). Satisfaction reduces to evaluation of the pricing predicates at x. The pricing lemmas establish boundedness of pstar and correctness of the deterministic pricing rule whenever p_has_cprice holds. The implication therefore holds pointwise for every j and lifts to G.

4.6 Verification vs. Satisfiability

This thesis does not search for traces satisfying a formula. It fixes the MUDA-induced trace from an initial state and proves in Rocq that this trace satisfies the fairness formulas. Since the transition system is deterministic, each initial state induces a unique infinite trace. The proofs establish that all such traces satisfy the fairness properties.

Therefore, the verification establishes that all executions generated by the MUDA transition system from initial_state inputs satisfy the specified fairness properties.

Thus, fairness is established for all executions generated by the MUDA transition system.
