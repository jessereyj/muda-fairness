EXECUTION ILLUSTRATION

5.1 Execution Scenario 1

Chapter 5 constructs a concrete execution trace generated by the deterministic transition function. It illustrates how the abstract state predicates and fairness properties are evaluated step by step along the trace. This chapter does not provide additional validation beyond the formal proofs in Chapter 4, but serves to make the semantics operational and observable.

Let sigma denote the infinite execution trace mu(x0) generated from x0 by repeated application of the deterministic transition function.

5.1.1 Initial Orders

Let the buyers and sellers be defined as follows.

Buyers:

b1 = (10, 5, 1)
b2 = (8, 4, 2)

Sellers:

s1 = (6, 3, 1)
s2 = (7, 4, 2)

The first component is the unit price. The second component is the initial unit quantity. The third component is the submission time.

Buyer priority:

b1 >_B b2, because 10 > 8

Seller priority:

s1 >_S s2, because 6 < 7

5.1.2 Initial State

The initial state x0 satisfies the following conditions.

phase(x0) = P1
residB(b1, x0) = 5
residB(b2, x0) = 4
residS(s1, x0) = 3
residS(s2, x0) = 4
M(x0) is empty
pstar is not yet determined

5.1.3 Execution Trace

Following the trace semantics defined in Chapter 4, the execution of the MUDA protocol is interpreted as an infinite trace. Each step of the execution corresponds to a position in the trace. The state at position i is obtained by applying the transition function i times from the initial state.

When the protocol reaches the terminal phase P7, the transition function returns the same state. The trace therefore continues by repeating this state. This follows the stuttering construction.

The evaluation of fairness properties at each step is based on the corresponding MUDA state. This follows the lifting of state predicates to trace positions.”

The execution trace proceeds through the seven protocol phases.

Note that the Phase P2 sorting transition produces a Phase P3 state before any match is executed.

The clearing price is computed during the transition from Phase P4 and is first stored in the state at Phase P5.

5.1.4 Matching Steps

The first feasible buyer-seller pair is (b1, s1).

b1 has the highest buyer priority.
s1 has the highest seller priority.
10 >= 6, so the pair is feasible.
traded quantity = min(5, 3) = 3

After this trade:

residB(b1) = 2
residS(s1) = 0

The next selected feasible pair is (b1, s2).

b1 still has higher priority than b2.
s2 is now the highest priority feasible seller.
10 >= 7, so the pair is feasible.
traded quantity = min(2, 4) = 2

After this trade:

residB(b1) = 0
residS(s2) = 2

The next selected feasible pair is (b2, s2).

b1 has no residual quantity, so it is no longer feasible.
b2 and s2 are feasible because 8 >= 7.
traded quantity = min(4, 2) = 2

After this trade:

residB(b2) = 2
residS(s2) = 0

No feasible buyer-seller pair remains. The protocol moves to P4.

5.1.5 Clearing Price

The last matched pair is (b2, s2, 2).

The final seller is fully matched, since residS(s2, x6) = 0. Therefore, the clearing price is determined from the buyer side. Hence,

pstar = p(b2) = 8

In the mechanization, this computation is performed by the Phase P4 pricing transition, and the resulting post-pricing state, where the clearing price field is set, is in Phase P5.

pstar = p(b2) = 8

5.1.6 Priority Fairness (Illustration)

Priority fairness is specified in Chapter 4 as an invariance property over the induced trace, an LTL G-formula. The general statement sigma, 0 |= priorityOK is established for all inputs by the Chapter 4 theorem, and Chapter 5 does not add validation beyond that.

Here we illustrate what the priority atoms mean on this concrete run:

Buyer side: while b1 remains feasible, the greedy selector chooses b1 ahead of b2.
Seller side: while s1 remains feasible for the chosen buyer, the selector chooses s1 ahead of s2.

These observations correspond to the step predicates evaluated along the Phase P3 portion of the trace.

5.1.7 Quantity Fairness (Illustration)

Quantity fairness is specified in Chapter 4 as an invariance property. At every time index, each agent's initial quantity equals allocated quantity plus residual quantity. The general LTL statement is proved in Chapter 4. Here we illustrate the accounting for Scenario 1.

For b1:

initial quantity = 5
traded quantity = 3 + 2 = 5
residual quantity = 0

Thus:

5 = 5 + 0

For b2:

initial quantity = 4
traded quantity = 2
residual quantity = 2

Thus:

4 = 2 + 2

For s1:

initial quantity = 3
traded quantity = 3
residual quantity = 0

Thus:

3 = 3 + 0

For s2:

initial quantity = 4
traded quantity = 2 + 2 = 4
residual quantity = 0

Thus:

4 = 4 + 0

The match record M and the residual quantities fully account for each agent's initial quantity in this execution.

5.1.8 Price Fairness (Uniform Pricing Illustration)

Price fairness is specified in Chapter 4 as an invariance property relating the existence of a clearing price to boundedness by the marginal pair and correctness of the deterministic pricing rule. The general LTL statement is proved in Chapter 4. Here we illustrate the concrete pricing outcome in Scenario 1.

In this execution:

pstar = 8

The computed clearing price pstar is determined from the marginal pair and the residual condition according to the deterministic pricing rule. This illustrates the concrete value returned by the mechanized determine_clearing_price function and, after pricing, the value stored in the state's clearing_price field.
