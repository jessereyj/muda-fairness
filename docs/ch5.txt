CHAPTER 5

EXECUTION SCENARIOS AND VALIDATION

5.1 Execution Scenario 1

This chapter presents concrete executions of the MUDA protocol and illustrates the satisfaction of the fairness properties defined in Chapter 4. Each execution scenario is represented as a sequence of states generated by the deterministic transition function. The state sequence shows how residual quantities, match records, and clearing price evolve during execution. The LTL fairness properties are then evaluated over the resulting execution trace.

The general correctness of these properties is established by the formal proofs in Chapter 4.

Let σ denote the infinite execution trace generated from x0 by repeated application of the deterministic transition function.

5.1.1 Initial Orders

Let the buyers and sellers be defined as follows.

Buyers:

b1 = (10, 5, 1)
b2 = (8, 4, 2)

Sellers:

s1 = (6, 3, 1)
s2 = (7, 4, 2)

The first component is the unit price. The second component is the initial unit quantity. The third component is the submission time.

Buyer priority:

b1 >_B b2, because 10 > 8

Seller priority:

s1 >_S s2, because 6 < 7
5.1.2 Initial State

The initial state x0 satisfies the following conditions.

phase(x0) = P1
residB(b1, x0) = 5
residB(b2, x0) = 4
residS(s1, x0) = 3
residS(s2, x0) = 4
M(x0) = empty
pstar is not yet determined
5.1.3 Execution Trace

The execution trace proceeds through the seven protocol phases.

Note that the Phase P2 sorting transition produces a Phase P3 state before any match is executed.

State	Phase	Residual Quantities	Match Record	Clearing Price
x0	P1	b1=5, b2=4, s1=3, s2=4	empty	undefined
x1	P2	b1=5, b2=4, s1=3, s2=4	empty	undefined
x2	P3	b1=5, b2=4, s1=3, s2=4	empty	undefined
x3	P3	b1=2, b2=4, s1=0, s2=4	[(b1, s1, 3)]	undefined
x4	P3	b1=0, b2=4, s1=0, s2=2	[(b1, s1, 3), (b1, s2, 2)]	undefined
x5	P3	b1=0, b2=2, s1=0, s2=0	[(b1, s1, 3), (b1, s2, 2), (b2, s2, 2)]	undefined
x6	P4	b1=0, b2=2, s1=0, s2=0	same as x5	undefined
x7	P5	b1=0, b2=2, s1=0, s2=0	same as x5	8
x8	P6	b1=0, b2=2, s1=0, s2=0	same as x5	8
x9	P7	b1=0, b2=2, s1=0, s2=0	same as x5	8

After x9, the execution trace continues by repeating x9.

5.1.4 Matching Steps

The first feasible buyer-seller pair is (b1, s1).

b1 has the highest buyer priority.
s1 has the highest seller priority.
10 >= 6, so the pair is feasible.
traded quantity = min(5, 3) = 3

After this trade:

residB(b1) = 2
residS(s1) = 0

The next feasible pair is (b1, s2).

b1 still has higher priority than b2.
s2 is now the highest priority feasible seller.
10 >= 7, so the pair is feasible.
traded quantity = min(2, 4) = 2

After this trade:

residB(b1) = 0
residS(s2) = 2

The next feasible pair is (b2, s2).

b1 has no residual quantity, so it is no longer feasible.
b2 and s2 are feasible because 8 >= 7.
traded quantity = min(4, 2) = 2

After this trade:

residB(b2) = 2
residS(s2) = 0

No feasible buyer-seller pair remains. The protocol moves to P4.

5.1.5 Clearing Price

The last matched pair is (b2, s2, 2).

The final seller is fully matched, since residS(s2, x6) = 0. Therefore, the clearing price is determined from the buyer side. Hence,

pstar = p(b2) = 8

In the mechanization, this computation is performed by the Phase P4 pricing transition, and the resulting post-pricing state (where the clearing price field is set) is in Phase P5.

This price is applied uniformly to all matched trades.

5.1.6 Validation of Priority Fairness

Priority fairness requires that a lower priority feasible agent is not matched ahead of a higher priority feasible agent.

In this execution:

b1 is matched before b2 while b1 remains feasible.
s1 is matched before s2 while s1 remains feasible.
no lower priority feasible buyer or seller is selected ahead of a higher priority feasible one.

Therefore, the priority condition holds at every matching step.

Let phi_PF denote the LTL formula for Priority Fairness. Then:

σ, 0 ⊨ phi_PF

5.1.7 Validation of Quantity Fairness

Quantity fairness requires that the initial quantity of each agent is equal to the sum of traded quantity and residual quantity.

For b1:

initial quantity = 5
traded quantity = 3 + 2 = 5
residual quantity = 0

So:

initial = allocated + residual
5 = 5 + 0

For b2:

initial quantity = 4
traded quantity = 2
residual quantity = 2

So:

4 = 2 + 2

For s1:

initial quantity = 3
traded quantity = 3
residual quantity = 0

So:

3 = 3 + 0

For s2:

initial quantity = 4
traded quantity = 2 + 2 = 4
residual quantity = 0

So:

4 = 4 + 0

Therefore, quantity is preserved for all agents throughout the execution.

Let phi_QF denote the LTL formula for Quantity Fairness. Then:

σ, 0 ⊨ phi_QF

5.1.8 Validation of Price Fairness

Price fairness requires that all matched trades are executed at one uniform clearing price.

In this execution:

clearing price = 8

All matched trades are executed at the clearing price p*. 
This satisfies the condition required by the LTL formula for price fairness, namely that every matched trade uses a uniform price.

Let phi_PrF denote the LTL formula for Price Fairness. Then:

σ, 0 ⊨ phi_PrF

5.1.9 Result

Execution Scenario 1 satisfies all three fairness properties.

Priority Fairness holds.
Quantity Fairness holds.
Price Fairness holds.

This scenario shows how the MUDA protocol generates a deterministic execution trace and how the fairness properties are evaluated over that trace.

This result is consistent with the general fairness theorems proven in Chapter 4, which establish that all execution traces generated by the deterministic transition system satisfy these properties.
