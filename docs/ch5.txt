CHAPTER 5

EXECUTION ILLUSTRATION

5.1 Execution Scenario 1

Chapter 5 constructs a concrete execution trace generated by the deterministic transition function. It illustrates how the abstract state predicates and fairness properties are evaluated step by step along the trace. This chapter does not provide additional validation beyond the formal proofs in Chapter 4, but serves to make the semantics operational and observable.

This chapter illustrates the theorems; it does not establish general correctness.

Chapter 5 does not introduce new results. It makes the semantics defined in Chapter 4 explicit and observable on a concrete execution.

Let sigma denote the infinite execution trace mu(x0) generated from x0 by repeated application of the deterministic transition function.

5.1.1 Initial Orders

Let the buyers and sellers be defined as follows.

Buyers:

b1 = (10, 5, 1)
b2 = (8, 4, 2)

Sellers:

s1 = (6, 3, 1)
s2 = (7, 4, 2)

The first component is the unit price. The second component is the initial unit quantity. The third component is the submission time.

Buyer priority:

b1 >_B b2, because 10 > 8

Seller priority:

s1 >_S s2, because 6 < 7

5.1.2 Initial State

The initial state x0 satisfies the following conditions.

phase(x0) = P1
residB(b1, x0) = 5
residB(b2, x0) = 4
residS(s1, x0) = 3
residS(s2, x0) = 4
M(x0) is empty
pstar is not yet determined

5.1.3 Execution Trace

Following the trace semantics defined in Chapter 4, the deterministic execution from x0 is interpreted as an infinite trace. For this scenario, we present the finite execution states
xi = execute i x0.

Each xi corresponds to position i in the trace mu(x0). This execute-to-trace correspondence is the lifting step from Chapter 4. It provides the lifting from state-level predicates to trace-level evaluation.

The execution trace proceeds through the seven protocol phases.

Note that the Phase P2 sorting transition produces a Phase P3 state before any match is executed.

The clearing price is computed during the transition from Phase P4 and is first stored in the state at Phase P5.

Each row shows one state xi = execute i x0, including phase, residual quantities, match record, and clearing price.

State | Phase | Residual Quantities | Match Record | Clearing Price
x0 | P1 | b1=5, b2=4, s1=3, s2=4 | empty | undefined
x1 | P2 | b1=5, b2=4, s1=3, s2=4 | empty | undefined
x2 | P3 | b1=5, b2=4, s1=3, s2=4 | empty | undefined
x3 | P3 | b1=2, b2=4, s1=0, s2=4 | [(b1, s1, 3)] | undefined
x4 | P3 | b1=0, b2=4, s1=0, s2=2 | [(b1, s1, 3), (b1, s2, 2)] | undefined
x5 | P3 | b1=0, b2=2, s1=0, s2=0 | [(b1, s1, 3), (b1, s2, 2), (b2, s2, 2)] | undefined
x6 | P4 | b1=0, b2=2, s1=0, s2=0 | same as x5 | undefined
x7 | P5 | b1=0, b2=2, s1=0, s2=0 | same as x5 | pstar = 8
x8 | P6 | b1=0, b2=2, s1=0, s2=0 | same as x5 | pstar = 8
x9 | P7 | b1=0, b2=2, s1=0, s2=0 | same as x5 | pstar = 8

Since x9 is in the terminal phase P7, the transition function satisfies delta(x9) = x9. The trace therefore continues by repeating x9, which is the terminal stuttering construction from Chapter 4.

After x9, the execution trace continues by repeating x9.

5.1.4 Matching Steps

The matching steps occur in Phase P3 (states x2 through x5 in the table).

From x2, the selector chooses the feasible pair (b1, s1). The traded quantity is
q = min(residB(b1, x2), residS(s1, x2)) = min(5, 3) = 3,
This produces x3, where residB(b1, x3) = 2 and residS(s1, x3) = 0.

From x3, the selector chooses (b1, s2). The traded quantity is
q = min(residB(b1, x3), residS(s2, x3)) = min(2, 4) = 2,
This produces x4, where residB(b1, x4) = 0 and residS(s2, x4) = 2.

From x4, the selector chooses (b2, s2). The traded quantity is
q = min(residB(b2, x4), residS(s2, x4)) = min(4, 2) = 2,
This produces x5, where residB(b2, x5) = 2 and residS(s2, x5) = 0.

No feasible buyer-seller pair remains at x5, so the execution advances to Phase P4 (x6).

5.1.5 Clearing Price

The last matched pair is (b2, s2, 2).

The marginal pair determines which side defines the price.

The final seller is fully matched, since residS(s2, x6) = 0. Therefore, the clearing price is determined from the buyer side. Hence,

pstar = p(b2) = 8

In the mechanization, this computation is performed by the Phase P4 pricing transition, and the resulting post-pricing state, where the clearing price field is set, is in Phase P5.

5.1.6 Priority Fairness (Illustration)

Priority fairness is specified and proved in Chapter 4 as an invariance property over the induced trace. Chapter 5 does not add validation beyond that proof; it illustrates how the priority atoms manifest on this concrete run.

The priority atoms mean the following on this concrete run:

Buyer side: while b1 remains feasible, the greedy selector chooses b1 ahead of b2.
Seller side: while s1 remains feasible for the chosen buyer, the selector chooses s1 ahead of s2.

These observations correspond to the step predicates evaluated along the Phase P3 portion of the trace.

These conditions can be verified at each matching step x2, x3, and x4.

5.1.7 Quantity Fairness (Illustration)

Quantity fairness is specified and proved in Chapter 4 as an invariance property. Here we illustrate the accounting identity on Scenario 1.

The same accounting identity holds at every state of the execution trace.

For b1:

initial quantity = 5
traded quantity = 3 + 2 = 5
residual quantity = 0

Thus:

5 = 5 + 0

For b2:

initial quantity = 4
traded quantity = 2
residual quantity = 2

Thus:

4 = 2 + 2

For s1:

initial quantity = 3
traded quantity = 3
residual quantity = 0

Thus:

3 = 3 + 0

For s2:

initial quantity = 4
traded quantity = 2 + 2 = 4
residual quantity = 0

Thus:

4 = 4 + 0

The match record M and the residual quantities fully account for each agent's initial quantity in this execution.

5.1.8 Price Fairness (Uniform Pricing Illustration)

Price fairness is specified and proved in Chapter 4 as an invariance property relating the existence of a clearing price to boundedness by the marginal pair and correctness of the deterministic pricing rule. Here we illustrate the concrete pricing outcome in Scenario 1.

In this execution:

pstar = 8

The computed clearing price pstar is determined from the marginal pair and the residual condition according to the deterministic pricing rule. This illustrates the concrete value returned by the mechanized determine_clearing_price function and, after pricing, the value stored in the state's clearing_price field.

This corresponds to the condition evaluated under the invariant
G (p_has_cprice -> (p_bounds_pstar /\ p_price_rule)).

This execution demonstrates how state-level predicates, phase transitions, and deterministic rules together produce an execution trace that satisfies the fairness invariants defined and formally verified in Chapter 4.
